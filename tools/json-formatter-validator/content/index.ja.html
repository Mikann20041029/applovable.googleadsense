結論（Quick answer）

JSONが通らない原因の大半は、末尾カンマ・キーのダブルクォート忘れ・シングルクォート・括弧の対応ミスです。 左に貼り付けて 検証 を押し、エラー（多くの場合は行/列つき）を見て修正し、最後に 整形（読みやすく）または 最小化（軽量化）を使って仕上げます。

このツールの使い方

  1. 入力欄にJSONを貼り付けます。迷ったら サンプル を押して、正しいJSONを入れてから試します。
  2. 検証 を押し、strict JSONとして正しいか判定します。誤りがあればメッセージと行/列（可能な場合）を表示します。
  3. 整形 で読みやすい形にし、最小化 で空白を消してサイズを減らせます。
  4. キーを並び替える は、文字列比較をするシステムでは差分が大きくなるので、必要なときだけ使います。
  5. コピー / ダウンロード で結果を保存できます。入力はブラウザ内に自動保存されます。

プライバシー：処理はブラウザ内だけで完結し、入力はサーバーへ送信されません。

「有効なJSON」とは何か（重要）

ネット上の“JSONっぽい”例は、実はstrict JSONではないことがよくあります。strict JSONは仕様（RFC 8259 / ECMA-404）に従います。 ここを理解すると、APIデバッグや設定ファイルで毎回同じミスを繰り返さなくなります。

ルール：JSONの型は6つだけ

  • オブジェクト：{ "key": "value" }
  • 配列：[1, 2, 3]
  • 文字列："text"（ダブルクォートのみ）
  • 数値：0, -12.34, 1e6（有限）
  • 真偽値：true/false
  • null：null

undefined や NaN、Infinity、関数、new Date() はJSONではありません。

ルール：キーは必ずダブルクォート

NG: { name: "Alice" }
OK: { "name": "Alice" }

ルール：文字列はダブルクォート

NG: { "name": 'Alice' }
OK: { "name": "Alice" }

ルール：末尾カンマ禁止

NG:
{ "a": 1, }

OK:
{ "a": 1 }

ルール：コメント禁止

// や /* */ はJSONにありません。JSONC対応の環境以外は落ちます。

典型エラーの直し方（順番が大事）

  1. まず 検証。メッセージと位置（行/列）を見る。
  2. その位置の文字を確認（余計なカンマ、足りないクォート、括弧ミスが多い）。
  3. {} と [] の対応を確認。
  4. 1つ直したらすぐ再検証。まとめて直すと原因が分からなくなります。

Troubleshooting（10個以上）

  • 末尾カンマを消す（}/] の直前）。
  • シングルクォートをダブルクォートへ。
  • キーを必ずダブルクォートで囲む。
  • コメントを削除する（JSONCは別物）。
  • バックスラッシュは \\ にする。
  • 文字列内改行は \n にする。
  • コピー漏れで末尾の括弧が落ちていないか。
  • [] と {} を混同していないか。
  • NaN/Infinity を使っていないか。
  • UTF-8で保存されているか（別エンコーディングで崩れるケース）。

FAQ（10個以上）

  • 安全？：ブラウザ内処理のみで、入力は送信されません。
  • コメントは？：標準JSONは不可です。
  • 末尾カンマは？：不可です。
  • 整形と最小化の違い：読みやすさ vs サイズ。
  • キー並び替え：意味は変えませんが、文字列比較の差分は変わります。
  • Schema検証：ここは構文のみ。形の検証はJSON Schemaで別途。
  • 大きいID：数値だと丸められることがあるので文字列推奨。
  • 日付：JSONに日付型はないのでISO文字列推奨。
  • Content-Type：APIは application/json; charset=utf-8 が一般的。
  • Unexpected end：括弧/クォートが閉じていません。

用語集（10語以上）

  • Object：{} のキー/値。

  • UTF-8：一般的な文字コード。

  • Strict：仕様に厳密なパース。
