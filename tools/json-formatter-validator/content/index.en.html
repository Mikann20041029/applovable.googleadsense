Quick answer

If your JSON is failing, the fastest fix is usually one of these: remove trailing commas, quote all object keys, use double quotes (not single quotes), and make sure brackets/braces match. Paste your JSON on the left, click Validate, and the tool will report the error (often with line/column). Then click Format to produce a readable version or Minify to compress it for transport.

What “valid JSON” really means

A lot of “JSON-like” text on the internet is not actually valid JSON. Standard JSON is defined by formal specifications (RFC 8259 and ECMA-404), and strict parsers (like JSON.parse) enforce these rules. Understanding the rules saves time because the same mistakes repeat across API debugging, configuration files, and data exports.

Rule 1: JSON has only these data types

  • Object: { "key": "value" }
  • Array: [1, 2, 3]
  • String: "text" (double quotes only)
  • Number: 0, -12.34, 1e6 (finite numbers only)
  • Boolean: true / false
  • Null: null

If you see undefined, NaN, Infinity, function bodies, or dates written like new Date(), that’s JavaScript (or another language), not JSON.

Rule 2: Object keys must be strings in double quotes

This fails in strict JSON: 

{ name: "Alice" }

This is correct: 

{ "name": "Alice" }

Rule 3: Strings must use double quotes

Single quotes are common in examples, but strict JSON does not allow them: 

{ "name": 'Alice' }

Use: 

{ "name": "Alice" }

Rule 4: No trailing commas

Trailing commas are one of the top causes of “it looks fine but won’t parse”. This is invalid: 

{
  "a": 1,
}

So is this: 

[1, 2, 3,]

Fix by removing the final comma before } or ].

Rule 5: Comments are not allowed

JSON does not support // or /* */. Some editors support “JSONC” (JSON with comments), but APIs and strict parsers will reject it. If your toolchain requires comments, store them outside the JSON (docs, README), or use a format that explicitly supports comments.

Rule 6: Encoding and escaping matter

JSON is typically transmitted as UTF-8. Most issues come from accidentally breaking strings: unescaped newline characters, unescaped quotes, or invalid backslashes. A backslash starts an escape sequence. For example, Windows paths often break because \U or \n is read as an escape. If you need a literal backslash, escape it: "C:\\Users\\Alice\\file.txt".

Step-by-step fixes for common errors

When you see an error like “Unexpected token …”, don’t guess. Fix systematically:

  1. Validate first. Read the message and locate the position (line/column if provided).
  2. Check the character at that spot: is it an extra comma, missing quote, or wrong bracket?
  3. Confirm structure: every { must close with }, every [ with ].
  4. Fix one thing at a time, then validate again. Multiple changes at once makes debugging harder.

“Unexpected token o in JSON at position …”

This classic error often appears when you are parsing something that is already an object (not a string) in JavaScript, or when you accidentally feed the parser non-JSON text like ok, error, or an HTML page. In a tool like this, it usually means your JSON starts with a character that’s not valid JSON. Valid JSON must start with {, [, ", a digit/minus, true, false, or null.

“Unexpected string”

Often caused by missing commas between properties: 

{
  "a": 1
  "b": 2
}

Add a comma after 1.

“Unexpected end of JSON input”

This means the parser reached the end while still expecting something: a closing brace/bracket, a quote, or a value. Typical causes:

  • Missing } or ]
  • Unclosed string: "name": "Alice
  • Cut-off payload (copy/paste truncated)

Broken strings (quotes, newlines, backslashes)

If your JSON includes large text blocks, stack traces, or multi-line content, make sure newlines are escaped as \n inside strings, and that embedded quotes are escaped as \". If you paste raw logs directly inside a JSON string without escaping, it will fail.

Best practices that prevent future pain

Valid JSON is the minimum. For real systems, consistency and safety matter too:

Use consistent indentation

Pick 2 or 4 spaces and stay consistent across a repository. Consistent formatting reduces merge conflicts and makes diffs readable.

Keep numbers precise (especially IDs)

JavaScript (and many tools) use IEEE-754 double precision for numbers. Very large integers can lose precision. If you have IDs like 12345678901234567890, store them as strings: "id": "12345678901234567890". This prevents accidental rounding.

Don’t rely on object key order

In JSON semantics, object key order is not significant. Some systems preserve order, but you should not depend on it unless the consumer explicitly says so. If order matters, use arrays.

Prefer explicit nulls over missing keys (when contract matters)

For APIs and configs, decide whether “missing” and “null” are different. If they are, document the contract. If you always include the key but allow it to be null, downstream code can be simpler.

Keep JSON small and split when necessary

Giant JSON blobs are hard to debug. If you control the format, split the payload into smaller objects or files, and validate each part independently.

Use JSON Schema when structure matters

Syntax validation only answers “is this JSON?”. JSON Schema answers “is this the right shape?”: required fields, types, enumerations, min/max, and more. For serious integrations, schema validation catches bugs earlier. This tool focuses on strict JSON syntax; schema validation is a separate step.

Security & privacy notes

This tool is designed to run fully in the browser. That means your JSON stays on your device. Still, be careful with sensitive payloads: if your browser has extensions that read pages, they could access input fields. For highly sensitive data, use a clean browser profile and avoid unknown extensions.

Troubleshooting (10+)

  • Trailing comma: remove the comma before } or ].
  • Single quotes: replace ' with " for strings and keys.
  • Unquoted keys: change { name: "x" } to { "name": "x" }.
  • Comments present: remove // or /* */ blocks.
  • Unescaped backslash: escape as \\ inside strings.
  • Unescaped newline in string: use \n instead of an actual newline.
  • Cut-off JSON: ensure copy/paste includes the final closing bracket/brace.
  • Wrong bracket type: arrays use [], objects use {}.
  • Invalid numbers: JSON does not allow NaN or Infinity.
  • Encoding issues: prefer UTF-8 when saving JSON files; avoid exotic encodings.

FAQ (10+)

Is this tool private?
Yes. JSON formatting and validation run locally in your browser. No input is sent to a server.
Why can’t I use comments?
Standard JSON has no comment syntax. Some environments support JSONC, but strict parsers will reject it.
Why do trailing commas fail?
Strict JSON disallows trailing commas. Remove them before } or ].
What’s the difference between Format and Minify?
Format adds whitespace/indentation for readability. Minify removes whitespace to reduce size.
Does sorting keys change my data?
It should not change meaning, but it changes raw text. Some systems compare raw JSON strings, so use with care.
Can I validate against JSON Schema here?
This tool validates strict JSON syntax. Schema validation is a separate step and is planned as a related tool.
Why does my ID number change?
Large integers can lose precision in some tools. Store large IDs as strings.
Can JSON contain dates?
JSON has no date type. Use ISO 8601 strings (e.g., "2026-01-27T12:34:56Z") and document it.
What content type should I use for APIs?
Typically Content-Type: application/json; charset=utf-8.
Why do I see “Unexpected end of JSON input”?
Your JSON is incomplete: missing a closing bracket/brace or an unclosed string.

Glossary (10+)

  • JSON: JavaScript Object Notation, a text format for data exchange.
  • Object: Key-value mapping in braces {}.
  • Array: Ordered list in brackets [].
  • String: Text value in double quotes.
  • Escape sequence: Backslash-based encoding like \n, \", \\.
  • Minify: Remove whitespace to reduce size.
  • Pretty print: Add whitespace and indentation for readability.
  • Schema: A formal definition of expected JSON structure (JSON Schema).
  • UTF-8: Common encoding used for JSON text.
  • Trailing comma: Extra comma after the last element/property (invalid in JSON).
